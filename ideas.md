# ideas.md — инбокс идей ShikiUpdatesBot

Летучий слой между «искрой» и каноном. Сюда падают сырые мысли, что приходят
по ходу работы и ещё никуда не созрели. Жизненный цикл записи:

  искра → дозрела → переезжает в один из домов:
    • GitHub issue        — задача («что сделать»)
    • custom instructions — процессное решение
    • AGENTS.md           — архитектурный факт «как устроено и почему»
    • секция ⛔ ниже      — отклонённая/отложенная идея (с причиной, паркуется здесь)

Не свалка и не дубль issues/инструкций. Старшинство правды:
инструкции/память проекта > этот файл > AGENTS.md (последний вторичен).

Файл версионируется в репозитории и синхронизируется между машинами через git.

## 🌱 Идеи (инбокс)

- **Poster carousels.** Карусели постеров в отчётах; начать с квартального
  топ-3. Сильно позже. Идея не финализирована — issue не заводим, пока не
  согласуем дизайн.
  Контекст: карусели/коллажи появились в Telegram Bot API 10.1 (Rich Messages,
  11.06.2026) — та же волна, что и rich-formatting для ветки refactor-messages.
  Дока: https://core.telegram.org/bots/api-changelog

- **Сборка в .exe.** Когда-то обсуждали standalone-исполняемый файл
  (PyInstaller/Nuitka) — запуск без установки Python, «двойной клик».
  Шальная идея, не финализирована. На подумать: где на десктопе хранить
  DATA_DIR и токен, автозапуск/фоновый режим, размер бандла, кроссплатформа.
  Issue не заводим, пока не согласуем смысл и дизайн.

- **Единое меню-«домашняя страница» через /start.** Объединить все команды
  (в т.ч. скрытые; их станет совсем много после команд «весь список
  просмотренного/прочитанного») в одно меню. Остаётся одна команда — /start:
  для новых инициирует общение, для всех вызывает домашнюю страницу
  (приветствие + основные действия кнопками) и удаляет мусор (саму /start
  пользователя). По выбору пункта сообщение динамически меняется под текущий
  шаг и удаляется по достижении конечной точки (напр. вывода статистики);
  дальше всё сначала через /start. Опционально — картинка в едином стиле на
  каждый шаг. Сложность: средняя. Не финализировано, issue не заводим.

- **Многопользовательский режим «я слежу за друзьями».** Сейчас режим «друзья
  следят за мной» (один профиль). Идея: бот проверяет несколько профилей.
  Добавляется в общий чат, настраивается и сообщает туда об активности
  участников (чьи профили добавлены); либо владелец добавляет профили друзей и
  получает уведомления сам. Многопользовательность должна работать со всеми
  функциями: уведомления, статистика, избранное и т.д. Сложность: сложное/
  очень сложное/возможно невыполнимое — помнить про лимиты API Shikimori.
  Не финализировано, issue не заводим.

- **Склонение DISPLAY_NAME по падежам.** Имя ({n}) вставляется в шаблоны как
  есть (именительный), но русский требует падежей: «зацепило Петя» → должно
  быть «Петю» (винительный). Проблема общая для всего банка сообщений.
  Варианты: (1) морфо-библиотека (petrovich — под русские имена с указанием
  пола; pymorphy3 тяжелее и по именам капризнее) — авто, мало конфига, но
  эвристика спотыкается на никах вроде «Ворга»; (2) ручные падежные формы в
  config (один юзер → 6 форм один раз; шаблоны берут {n_acc}/{n_dat}/…, фолбэк
  на именительный) — 100% верно, без зависимостей, но надо тегировать падеж в
  каждом шаблоне; (3) переписать шаблоны под именительный — без зависимостей,
  но душит стиль. Лин: гибрид — petrovich авто + ручные оверрайды на редкие
  косяки + фолбэк на именительный. Делать ПОСЛЕ рефактора (банк уедет в
  messages.py — там тегировать падежи удобнее). Не финализировано, issue не
  заводим.

- **Бэкапы в GitHub Gist (альтернативный канал хранения стейта).** Сейчас у
  бэкапа один канал — личка владельца; если владелец недоступен/заблокировал
  бота, на эфемерном хосте стейт теряется (см. owner-gate в 🧭). Идея:
  дублировать бэкап-архив (или его содержимое) в приватный GitHub Gist как
  независимый канал, не завязанный на доступность Telegram-владельца.
  Обсуждали в другом чате — детали (auth-токен, приватность, размер, ротация
  Gist) восстановить оттуда при возврате. Не финализировано, issue не заводим.

- **Латинские гомоглифы в парсерах истории Шикимори.** score-change-fix
  вскрыл: Шикимори шлёт латинскую «c» (U+0063) вместо кириллической «с» в
  «изменена оценка c X на Y» — регулярка мимо, оценка рендерилась как «?».
  Точечно вылечили один коннектив (класс [сc]), но та же мина возможна в
  любом кириллическом слове с латинским двойником (на, о, е, р, х, у…) по
  всему банку (classify_event, extract_score, extract_score_change). Идея:
  единая нормализация латинских гомоглифов → кириллица ПЕРЕД матчингом —
  но аккуратно с англ. форматами (rated/scored), чтобы их не сломать.
  Взвесить: точечные [xy]-классы против одной точки нормализации. Не
  финализировано, issue пока не заводим.

- **Два банка для score_changed: оценка выросла / упала.** Сейчас смена оценки —
  один нейтральный банк, тон не зависит от направления. После score-change-fix
  extract_score_change надёжно отдаёт (old, new) → направление известно даром.
  Идея: развести на два банка с тоном — рост (одобрение, «пересмотрел и
  проникся») и падение (разочарование, «остыл»). NB: для score_changed old≠new
  всегда (иначе не событие) — нейтрального остатка нет, два банка кроют всё.
  Взвесить калибровку тона, чтобы падение не съезжало в токсичность. Не
  финализировано, issue пока не заводим.

## 🧭 Архитектурные заметки на будущее

- **Общий FSM-каркас (fsm_common / bot_utils).** Если понадобится ещё
  разгрузить handlers.py — вынести вниз общие хелперы FSM-обёрток:
  _safe_delete (генерик-чистка сообщений; его зовут и broadcast, и backup, и
  cmd_cancel), возможно _confirm_kb. Тогда тонкие обёртки broadcast/backup
  перестанут тянуть их из handlers — реальная развязка. Это, а не broadcast.py,
  — правильный следующий шаг дробления, если он вообще понадобится.


- **owner-gate — только на старте; рантайм-фейлы деградируют мягко.**
  probe_owner_and_start проверяет доступность владельца ОДИН раз при старте
  (доставилось 🟢 → цикл; нет → WARNING, цикл не стартуем, /start добудит).
  В рантайме доступность НЕ перепроверяется — намеренно: остановить цикл из-за
  владельца = лишить уведомлений остальных подписчиков. Если владелец блокирует
  бота позже: цикл живёт; send_to_all_chats сам отписывает заблокировавшего
  владельца (как любого); бэкапы/ошибки/отчёты владельцу фейлятся-и-логируются
  под своими guard'ами, вызывающий флоу не падает. Единственная реальная потеря
  — недоставленные бэкапы (на эфемерных хостах = потеря стейта; канал один,
  митигации нет; self-inflicted). Самолечение: на следующем рестарте проба не
  доставится → цикл не стартует. Рантайм-ре-гейтинг не делаем (net-negative).

## ⛔ Отклонено / отложено (с причиной)

Идеи, которые сознательно отвергли. Не обязательно навсегда — что не нужно
сейчас, может пригодиться, когда бот вырастет. У каждой записи — конкретная
причина: вернёшься к идее → перечитай, почему зарубили, и проверь, не изменился
ли контекст. Не перепредлагать без новой причины.

- **«Problem B» — бэкафилл stats_current из истории.**
  Причина: отклонено по итогам обсуждения; детали в инструкции не зафиксированы —
  при возврате восстановить из истории чата (впредь причину пишем сразу).
- **Джиттер интервала поллинга.**
  Причина: фиксированный интервал намеренно (firewall-философия) — предсказуемый
  ритм важнее размазывания нагрузки и рестартов.
- **Оценка 4 в отдельный/низкий банк оценок.**
  Причина: остаётся в mid. Без выделенного бакета четвёрку чисто не отделить, а
  ради редкого случая отдельный бакет не оправдан.

- **Выделение /broadcast в отдельный модуль (broadcast.py).**
  Причина: нечего выносить. В отличие от backup (толстая логика: zip/restore/
  валидаторы/триггеры/send_backup), у broadcast вся «логика» — одна функция
  _send_broadcast_message; остальное тонкие FSM-обёртки (cmd_broadcast,
  broadcast_receive/confirm/cancel), намертво сидящие на общем каркасе
  (_safe_delete — генерик; cmd_cancel — одна команда на оба FSM). Вынос дал бы
  тот же цикл handlers↔broadcast, что разруливали у бэкапа, но без выгоды.
  NB: send_to_all_chats — notification-домен (её зовёт check_and_notify), не
  broadcast, так что с ним бы и не уехала. Нужна разгрузка handlers — см.
  fsm_common в 🧭, не broadcast.py.

- **Схлопывание «избыточного» init-блока polling_loop.**
  Причина: после boot-throttle перестало быть избыточным. init теперь несёт
  реальный смысл — общая ClientSession на старте + фетч favourites один раз для
  sync(fav=); схлопывание увело бы первый baseline в цикловые check_and_notify*
  (каждая со своей сессией) → менее throttled старт + двойной фетч favourites.
  А baseline в check_and_notify* — не дубль, а намеренная 429-резильентность
  (fallback, если стартовый фетч упал из-за 429/сети). Слоистость init=primary /
  check=fallback оставляем. Дедуп-фолд закрыт частью 1 (общий хелпер отписки).

- **Единый маркер «последнего бэкапа вообще» (учёт аварийного/подписочного).**
  Причина: last_backup_at живёт в stats_current, которым владеет in-memory cur
  в polling-цикле; честный учёт событийных бэкапов (shutdown/под-отписка)
  потребовал бы разруливать это владение (storage-хелпер на одно поле или вынос
  метки в отдельный файл) + тесты — сложность без сопоставимой пользы. Событийные
  бэкапы владелец и так видит вживую (каждый — сообщение с #backup). Снапшот
  сигналит о ПЕРИОДИЧЕСКОЙ страховке → подпись честно сузили до «Последний
  плановый бэкап» (стемпят только еженедельный + ротация квартала). Вернуться —
  только если появится хост без live-доставки владельцу.

- **Форс пустых строк через ruff E3 (E301–E306).**
  Причина: в нашей версии ruff эти правила preview-gated («has no effect because
  preview is not enabled») — включение требует preview=true, а это тянет пачку
  нестабильных preview-правил ради одних пустых строк — непропорционально.
  Стабильный путь к «воздуху» — ruff format (сам ставит 2 строки между top-level
  def), но это репо-широкий реформат = отдельное решение/ветка, не сейчас. Пока
  «воздух в коде» — на человеческий глаз. NB: дозапись-в-конец-файла у Claude
  однажды съедала ведущие пустые строки — следить вручную.
