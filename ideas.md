# ideas.md — инбокс идей ShikiUpdatesBot

Летучий слой между «искрой» и каноном. Сюда падают сырые мысли, что приходят
по ходу работы и ещё никуда не созрели. Жизненный цикл записи:

  искра → дозрела → переезжает в один из домов:
    • GitHub issue        — задача («что сделать»)
    • custom instructions — процессное решение
    • AGENTS.md           — архитектурный факт «как устроено и почему»
    • секция ⛔ ниже      — отклонённая/отложенная идея (с причиной, паркуется здесь)

Не свалка и не дубль issues/инструкций. Старшинство правды:
инструкции/память проекта > этот файл > AGENTS.md (последний вторичен).

Файл версионируется в репозитории и синхронизируется между машинами через git.

## 🌱 Идеи (инбокс)

- **Poster carousels.** Карусели постеров в отчётах; начать с квартального
  топ-3. Сильно позже. Идея не финализирована — issue не заводим, пока не
  согласуем дизайн.
  Контекст: карусели/коллажи появились в Telegram Bot API 10.1 (Rich Messages,
  11.06.2026) — та же волна, что и rich-formatting для ветки refactor-messages.
  Дока: https://core.telegram.org/bots/api-changelog

- **Сборка в .exe.** Когда-то обсуждали standalone-исполняемый файл
  (PyInstaller/Nuitka) — запуск без установки Python, «двойной клик».
  Шальная идея, не финализирована. На подумать: где на десктопе хранить
  DATA_DIR и токен, автозапуск/фоновый режим, размер бандла, кроссплатформа.
  Issue не заводим, пока не согласуем смысл и дизайн.

- **Аудит/чистка тестов (отдельная миниветка).** Тесты писал ChatGPT давно,
  не факт что оптимально: возможны дубли, слабые ассерты (проходят на сломанном
  коде), оверммокинг, мёртвые тесты, дыры в покрытии. Дисциплина: правки
  контента тестов НЕ мешать с механическим сплитом — разные ветки. Критерий
  аудита: каждый тест обязан падать на непропатченном коде (проверять мутацией).
  Открытый вопрос ПОРЯДКА: глубокий аудит после рефактора (чистим по модулям),
  но лёгкий триаж-«доверяй сети» — ДО рефактора (крепче страховка под страшную
  ветку). Текущий план: триаж до → рефактор → глубокий аудит после.

- **Единое меню-«домашняя страница» через /start.** Объединить все команды
  (в т.ч. скрытые; их станет совсем много после команд «весь список
  просмотренного/прочитанного») в одно меню. Остаётся одна команда — /start:
  для новых инициирует общение, для всех вызывает домашнюю страницу
  (приветствие + основные действия кнопками) и удаляет мусор (саму /start
  пользователя). По выбору пункта сообщение динамически меняется под текущий
  шаг и удаляется по достижении конечной точки (напр. вывода статистики);
  дальше всё сначала через /start. Опционально — картинка в едином стиле на
  каждый шаг. Сложность: средняя. Не финализировано, issue не заводим.

- **Многопользовательский режим «я слежу за друзьями».** Сейчас режим «друзья
  следят за мной» (один профиль). Идея: бот проверяет несколько профилей.
  Добавляется в общий чат, настраивается и сообщает туда об активности
  участников (чьи профили добавлены); либо владелец добавляет профили друзей и
  получает уведомления сам. Многопользовательность должна работать со всеми
  функциями: уведомления, статистика, избранное и т.д. Сложность: сложное/
  очень сложное/возможно невыполнимое — помнить про лимиты API Shikimori.
  Не финализировано, issue не заводим.

## ⛔ Отклонено / отложено (с причиной)

Идеи, которые сознательно отвергли. Не обязательно навсегда — что не нужно
сейчас, может пригодиться, когда бот вырастет. У каждой записи — конкретная
причина: вернёшься к идее → перечитай, почему зарубили, и проверь, не изменился
ли контекст. Не перепредлагать без новой причины.

- **«Problem B» — бэкафилл stats_current из истории.**
  Причина: отклонено по итогам обсуждения; детали в инструкции не зафиксированы —
  при возврате восстановить из истории чата (впредь причину пишем сразу).
- **Джиттер интервала поллинга.**
  Причина: фиксированный интервал намеренно (firewall-философия) — предсказуемый
  ритм важнее размазывания нагрузки и рестартов.
- **Оценка 4 в отдельный/низкий банк оценок.**
  Причина: остаётся в mid. Без выделенного бакета четвёрку чисто не отделить, а
  ради редкого случая отдельный бакет не оправдан.
