# 🎌 [ShikiUpdatesBot](https://github.com/WorgaNomoR/ShikiUpdatesBot) — Shikimori Activity Tracker

[![Tests](https://github.com/WorgaNomoR/ShikiUpdatesBot/actions/workflows/tests.yml/badge.svg?branch=main)](https://github.com/WorgaNomoR/ShikiUpdatesBot/actions)

![Docker](https://img.shields.io/badge/-Docker-2496ED?style=flat&logo=docker&logoColor=white)
[![License: GPL v3](https://img.shields.io/badge/License-GPLv3-blue.svg)](https://www.gnu.org/licenses/gpl-3.0)

Telegram-бот, который следит за аниме и мангой любого пользователя на [Shikimori](https://shikimori.io) и отправляет подписчикам весёлые уведомления о каждом событии: что человек начал смотреть, бросил, оценил или добавил в избранное.

Достаточно указать ник отслеживаемого пользователя — и бот готов к работе (отображаемое имя можно задать отдельно, но не обязательно). Никаких ключей API и авторизаций: всё работает на публичных данных Shikimori.

## Что умеет

- 📡 Проверяет историю и избранное пользователя каждые **15 минут**
- 🎬 Реагирует на события: добавил в список, начал смотреть/читать, пересматривает/перечитывает, бросил, завершил, поставил или изменил оценку
- ⭐ Отслеживает новые добавления в избранное: аниме, мангу, ранобэ, персонажей и людей индустрии
- ✅ Поддерживает **аниме** (TV, фильм, OVA, ONA) и **мангу** (кроме ваншотов и додзинси)
- 📋 Показывает, что отслеживаемый пользователь смотрит и читает прямо сейчас
- 📊 Собирает **статистику** просмотренного и прочитанного: жанры, темы, аудитория, студии, издатели, источники адаптации, годы выпуска, оценки и сравнение со средним рейтингом Shikimori
- 🗓️ В начале каждого квартала автоматически присылает владельцу **развёрнутый отчёт** с достижениями и сравнением с прошлым кварталом
- ❤️ Показывает избранное — любимые аниме, мангу, ранобэ, персонажей и людей индустрии, со ссылками и оценками
- 📢 Владелец может обратиться ко всем подписчикам одним сообщением
- 💾 Резервное копирование всех данных — автоматическое (при подписках/отписках, смене квартала, еженедельно и перед плановой остановкой) и ручное одной командой
- 👥 Система подписки — каждый подписывается сам командой `/start`
- 🩺 HTTP-эндпоинт `/health` для healthcheck — совместимость с хостингами и автоперезапуск при зависании
- ⚠️ При ошибках фоновой проверки пишет в лог и уведомляет владельца в Telegram (с антиспамом)

## Команды

| Команда | Описание |
|---|---|
| `/start` | Подписаться на уведомления |
| `/stop` | Отписаться |
| `/status` | Что отслеживаемый пользователь смотрит и читает прямо сейчас |
| `/stats` | Статистика — открывает меню с кнопками выбора отчёта |
| `/favs` | Избранное: любимые аниме, манга, ранобэ, персонажи и люди индустрии |
| `/broadcast` | Написать сообщение подписчикам *(только для владельца)* |
| `/cancel` | Отменить текущую операцию *(только для владельца)* |
| `/subs` | Список подписчиков *(только для владельца)* |
| `/backup` | Экспорт/импорт всего состояния архивом *(только для владельца)* |

## Статистика и отчёты

Бот ведёт статистику активности отслеживаемого пользователя и строит по ней наглядные отчёты с выровненными колонками, процентами и распределением оценок.

**Команда `/stats`** открывает меню с кнопками — можно выбрать статистику **за текущий квартал** или **за всё время**. Выбрал кнопку → пришёл отчёт, само меню удаляется, чтобы не захламлять чат. Есть и быстрый путь: `/stats all` сразу присылает статистику за всё время, минуя меню.

**Квартальный отчёт** формируется автоматически при смене квартала (1 января, апреля, июля, октября) и отправляется владельцу. Он включает полную статистику за квартал, сравнение с предыдущим и блок «достижений». Если бот был выключен в момент смены квартала — отчёт сформируется при следующем запуске (с защитой от повторной отправки).

**Что считается:** завершённое, брошенное и добавленное в планы, средняя оценка и её распределение, топ по оценкам, разбивка по жанрам / темам / аудитории, студии (для аниме) и издатели (для манги), источники адаптации, возрастные рейтинги, хронология по годам, примерное время просмотра и количество глав, а также сравнение личных оценок со средним рейтингом Shikimori.

**Откуда данные.** Списки пользователя (что просмотрено, оценки, прогресс) берутся из публичного экспорта Shikimori — без авторизации. Метаданные тайтлов (жанры, студии, год и т.д.) подтягиваются через GraphQL API батчами, только для новых тайтлов. Команды `/stats` и `/favs` читают готовые локальные файлы и не делают сетевых запросов — отвечают мгновенно и не зависят от доступности Shikimori.

При первом запуске в середине квартала статистика за текущий квартал будет неполной — об этом честно пишется в заголовке (`статистика с DD.MM.YYYY`). А статистика за всё время строится сразу и полностью, потому что берётся из экспорта списков, а не из накопленных событий.

## Избранное

**Команда `/favs`** показывает любимое — список избранного со ссылками на Shikimori и проставленными оценками (там, где они есть). Учитываются все категории избранного Shikimori: аниме, манга, ранобэ, персонажи и люди индустрии. Пустые категории не показываются.

Это намеренно отдельная от статистики история: в `/stats` — агрегаты и цифры, в `/favs` — то, что человек выделил как любимое. Поэтому избранное живёт в своей команде, а не в общем отчёте.

## Запуск

Бот можно запустить тремя способами — от самого простого к самому гибкому. Выбирай тот, что тебе по силам, остальные знать не обязательно.

Но сначала, в любом случае, понадобятся две вещи:

**Токен бота.** Напиши [@BotFather](https://t.me/BotFather) → команда `/newbot` → следуй инструкциям → получи токен (длинная строка вида `123456:ABC-DEF...`).

**Свой Telegram ID.** Напиши [@userinfobot](https://t.me/userinfobot) — он пришлёт твой числовой ID. Это нужно, чтобы бот знал, кто его владелец (админ-команды и уведомления об ошибках).

Эти два значения — `BOT_TOKEN` и `OWNER_ID` — понадобятся при любом способе запуска.

**Укажи, за кем следить.** По умолчанию бот настроен на профиль автора — `WNR`. Открой `main.py` и поменяй константу `SHIKI_USER` в начале файла на ник нужного пользователя с Shikimori — иначе бот будет присылать уведомления про чужой профиль. Имя в сообщениях бот возьмёт из этого же ника; при желании его можно переопределить переменной окружения `DISPLAY_NAME`.

**Подпишись на своего бота.** После первого запуска открой бота в Telegram и нажми `/start` — это обязательно, а не для удобства. Бот шлёт владельцу уведомления о нештатных ситуациях (сбой опроса Shikimori, ошибки в фоновом цикле), и доставить их он может только подписчику. Если владелец не подписан, аварийные сообщения уходят в никуда — ты узнаёшь о проблеме последним. Бонусом: будучи подписчиком, ты видишь свои же рассылки ровно так, как их получают остальные (при `/broadcast` отдельного превью нет by design — ты получаешь рассылку как обычный подписчик).

### Способ 1. На хостинге через GitHub (рекомендуется)

Самый простой путь, не требующий ни терминала, ни Docker, ни своего сервера. Подходит, даже если ты не программист — большинство платформ сделают всё за тебя.

Идея одна и та же на любой платформе:

1. Сделай **форк** этого репозитория к себе на GitHub (кнопка Fork вверху страницы). Затем открой в форке `main.py` и поменяй `SHIKI_USER` на ник пользователя, за которым нужно следить (по умолчанию — `WNR`, профиль автора).
2. Зарегистрируйся на хостинг-платформе и подключи к ней свой GitHub-форк.
3. Платформа увидит в репозитории `Dockerfile` и **сама соберёт и запустит** бота — тебе не нужно ничего про Docker знать.
4. В настройках проекта задай переменные окружения `BOT_TOKEN` и `OWNER_ID` (как их получить — см. выше).
5. Если платформа просит указать Healthcheck Path — впиши `/health` (подробнее в разделе [Healthcheck](#healthcheck)).
6. **Подключи постоянное хранилище (persistent volume / disk) с путём монтирования `/data`** — иначе состояние бота не переживёт редеплой (см. предупреждение ниже).

Готово — бот работает и будет перезапускаться сам.

> ⚠️ **Про сохранность данных — обязательно к прочтению.** Бот хранит подписчиков, статистику и кварталы в каталоге `/data`. По умолчанию это **эфемерная** папка внутри контейнера: при каждом редеплое (а на хостингах с автодеплоем — при каждом пуше в GitHub) она **стирается вместе с данными**. Чтобы состояние сохранялось, в настройках сервиса нужно примонтировать persistent volume / disk на путь `/data`. Если платформа этого не умеет (или только на платном тарифе) — данные будут теряться при каждом обновлении; тогда либо выбери платформу с поддержкой томов (например, Railway), либо регулярно делай резервную копию состояния через `/backup` и восстанавливай её после редеплоя. Самое невосстановимое здесь — список подписчиков (его ниоткуда не реконструировать), так что `/backup` перед обновлением — твоя страховка. Вдобавок бот сам присылает владельцу свежий архив при каждой подписке/отписке, смене квартала и раз в неделю — на эфемерном хосте это и страховка состояния, и зонд внутрь контейнера.

**Выбор платформы.** У каждой свои плюсы и подводные камни:

| Платформа | Плюсы | Минусы |
|---|---|---|
| [apply.build](https://apply.build) | Щедрый бесплатный тариф, простое подключение через GitHub | **Нет постоянного хранилища** — `/data` стирается при каждом редеплое, состояние (в т.ч. подписчики) не сохраняется (см. предупреждение выше); к тому же маловато настроек — бота, например, нельзя остановить из интерфейса |
| [Railway](https://railway.app) | Очень удобный, много настроек и возможностей; поддерживает постоянные тома (`/data` переживает редеплой) | Каждый месяц начисляется ~$1 бесплатных кредитов на ресурсы — активному боту их может не хватить до конца месяца; оплата сверх лимита — только **иностранной картой** |
| [bothost.ru](https://bothost.ru) | Российский (оплата работает из РФ), 99₽/мес за 5 ботов, проще VPS | Платный с самого начала |

> ⚠️ **Для пользователей из России:** иностранные платформы (Railway, Render, Fly.io и т.п.) сейчас нельзя оплатить картами, выпущенными в РФ. Если нужен платный тариф — смотри в сторону российских хостингов вроде bothost.ru или щедрого бесплатного apply.build.

Существуют и хостинги вроде JustRunMyApp, Pella, Tranger Cloud, но многие из них требуют вручную продлевать работу бота каждые 1–3 дня — для бота, который должен жить постоянно, это неудобно.

### Способ 2. Локально или на своём сервере

Подходит, если ты знаком с Python и хочешь погонять бота у себя на машине или на собственном сервере.

```bash
# 1. Клонировать репозиторий
git clone https://github.com/WorgaNomoR/ShikiUpdatesBot.git
cd ShikiUpdatesBot

# 2. Установить зависимости
pip install -r requirements.txt

# 3. Задать переменные окружения
export BOT_TOKEN="токен_от_BotFather"
export OWNER_ID="твой_telegram_id"

# 4. Запустить
python main.py
```

Перед первым запуском открой `main.py` и поменяй `SHIKI_USER` на ник пользователя, за которым следишь (по умолчанию там профиль автора — `WNR`).

Файл `.env` (см. `.env.example`) подхватывают `docker-compose` (способ 3) и большинство хостингов при импорте переменных. А локальный `python main.py` читает только переменные из окружения — для него используй `export` выше (автозагрузку `.env` для локального запуска добавим позже).

> 💡 Учти: при таком запуске бот живёт, только пока запущен процесс. Закрыл терминал или выключил компьютер — бот остановился. Для постоянной работы 24/7 нужен либо способ 1 (хостинг), либо свой всегда включённый сервер.

### Способ 3. Docker вручную

Подходит, если у тебя свой VPS и ты знаешь, что такое Docker. (Если не знаешь — спокойно используй способ 1, там Docker работает «под капотом» без твоего участия.)

Как и в других способах, перед сборкой поменяй `SHIKI_USER` в `main.py` на нужный ник.

```bash
# 1. Создай файл .env в корне проекта:
#    BOT_TOKEN=токен_от_BotFather
#    OWNER_ID=твой_telegram_id

# 2. Собери и запусти контейнер
docker-compose up -d --build

# 3. Логи
docker-compose logs -f

# 4. Остановить
docker-compose down
```

Для обновления после изменений в коде:

```bash
docker-compose up -d --build
```

Данные бота по умолчанию хранятся в `/data` внутри контейнера; в `docker-compose.yml` папка `./data` с хоста монтируется туда, так что состояние переживает пересборку. Если нужен доступ к healthcheck-эндпоинту снаружи контейнера — добавь в `docker-compose.yml` проброс порта:

```yaml
    ports:
      - "8080:8080"
```

## Healthcheck

Бот поднимает лёгкий HTTP-сервер параллельно с основным циклом (модуль `healthcheck.py`). Он нужен для двух целей:

- **Совместимость с хостингами**, которые требуют открытый порт или healthcheck-эндпоинт (Render, Fly.io, Cloud Run и подобные). Без него такие платформы могут считать, что процесс не запустился.
- **Watchdog**: эндпоинт `/health` отдаёт `200`, только если бот «пульсирует» — недавно успешно завершил итерацию фонового цикла. Если пульс протух (цикл завис или умер) — отдаётся `503`, и хостинг с настроенным healthcheck перезапустит контейнер.

| Эндпоинт | Ответ |
|---|---|
| `GET /health` | `200` если бот жив, `503` если последний успешный цикл был давно |
| `GET /` | `200` — простой признак, что порт открыт |

**Порог живости** адаптивный — три пропущенных цикла (`CHECK_INTERVAL × 3`, по умолчанию 45 минут). **Порт** берётся из переменной окружения `PORT` (хостинги обычно прокидывают её сами), по умолчанию `8080`. В настройках хостинга в качестве Healthcheck Path укажи `/health`.

**Настройки liveness-пробы.** Порог в 45 минут и подсчёт пропущенных циклов
уже внутри `/health`, поэтому проба платформы должна быть тонким частым
триггером, а не считать провалы сама:

| Параметр | Рекомендация |
|---|---|
| Interval | ~60 сек — **не** ставь большой интервал (40+ мин): проба проскочит переход в `503`, и бот пролежит вдвое дольше |
| Max failures | 1–2 (`503` уже значит «точно мёртв 45 минут») |
| Timeout | ~10 сек |
| Initial delay | ~15–20 сек |

«Сколько простоя до перезапуска» меняется в коде (`CHECK_INTERVAL × misses`),
а не увеличением интервала пробы.

Дополнительных зависимостей не требуется — используется серверная часть уже установленного `aiohttp`.

## Конфигурация

Основные параметры в начале `main.py`:

| Константа | По умолчанию | Описание |
|---|---|---|
| `BOT_TOKEN` | из переменной окружения | Токен Telegram-бота от BotFather |
| `OWNER_ID` | из переменной окружения | Telegram ID владельца для админ-команд и уведомлений об ошибках |
| `SHIKI_USER` | `WNR` | Ник отслеживаемого пользователя на Shikimori |
| `DISPLAY_NAME` | ник из `SHIKI_USER` | Имя в сообщениях; задаётся переменной окружения |
| `SHIKI_BASE_URL` | `https://shikimori.io` | Домен Shikimori — менять при смене зеркала |
| `CHECK_INTERVAL` | `900` (15 мин) | Интервал проверки в секундах |
| `ERROR_NOTIFY_INTERVAL` | `1800` (30 мин) | Минимальная пауза между уведомлениями владельцу об ошибках |
| `FULL_SYNC_INTERVAL` | `21600` (6 ч) | Как часто фоновый цикл пересинхронизирует полную статистику (`stats_all`) |
| `WEEKLY_BACKUP_INTERVAL` | `604800` (7 дн) | Период еженедельного авто-бэкапа состояния владельцу (по метке `last_backup_at`) |
| `DATA_DIR` | `/data` | Папка для файлов состояния; задаётся переменной окружения |
| `PORT` | `8080` | Порт healthcheck-сервера; задаётся переменной окружения |

Чтобы отслеживать другого пользователя — поменяй `SHIKI_USER` в начале `main.py`. Отображаемое имя бот возьмёт из ника автоматически; при желании переопредели его переменной окружения `DISPLAY_NAME`.

## Поведение при ошибках

Фоновый цикл проверки не останавливается из-за единичной ошибки Shikimori API, сети, парсинга или отправки сообщения. Бот:

- пишет полный traceback в лог;
- пробует отправить владельцу короткое сообщение об ошибке в Telegram;
- не шлёт такие уведомления чаще одного раза в `ERROR_NOTIFY_INTERVAL`;
- продолжает работу и повторяет проверку после `CHECK_INTERVAL`.

Сбор статистики устроен так же отказоустойчиво: недоступность экспорта или GraphQL не роняет бота — отчёт просто выйдет без обогащённых метаданных, а битый файл состояния заменяется пустой структурой. Команды `/stats` и `/favs` не делают сетевых запросов и не могут упасть из-за недоступности Shikimori.

Если процесс Python полностью завершился, сервер выключился или Telegram недоступен, уведомление об ошибке отправить уже не получится — в таких случаях помогут логи хостинга или внешний мониторинг (для последнего и пригодится healthcheck-эндпоинт).

## Файлы данных

Бот хранит файлы состояния — **их не нужно коммитить** (они уже в `.gitignore`):

- `seen_ids.json` — ID уже обработанных событий истории
- `subscribers.json` — список подписчиков
- `seen_favourites.json` — ID уже обработанных записей избранного
- `stats_all.json` — статистика за всё время: архив тайтлов, агрегаты и избранное
- `stats_current.json` — события текущего квартала
- `quarters/` — замороженные снапшоты завершённых кварталов

По умолчанию файлы создаются в каталоге `/data`. Чтобы хранить их в другом месте, укажи путь через переменную окружения:

```
DATA_DIR=/путь/к/папке
```

При первом запуске бот молча запоминает текущую историю и избранное и не спамит прошлыми событиями — отслеживает только новые.

## Резервное копирование и перенос состояния

Команда `/backup` (только для владельца) открывает меню с двумя кнопками:

- **📤 Экспорт** — бот присылает zip-архив всего каталога `/data`: подписчики, статистика за всё время, текущий квартал и снапшоты прошлых кварталов.
- **📥 Импорт** — пришли боту такой архив файлом, и он восстановит **подписчиков, текущий квартал и снапшоты кварталов**. Остальное (служебные `seen_*` и `stats_all`, которые бот и так пересобирает сам) из архива намеренно игнорируется.

**Перенос при смене хостинга:** на старом хостинге `/backup` → 📤 Экспорт, сохрани архив. На новом — запусти бота, затем `/backup` → 📥 Импорт и пришли архив.

> ⚠️ После выгрузки архива обязательно останови (или удали) бота на старом хостинге, прежде чем поднимать на новом. Два процесса с одним `BOT_TOKEN` начнут конфликтовать за `getUpdates` (Telegram отдаёт «terminated by other getUpdates request»), и оба будут работать нестабильно.

**Авто-бэкап.** Помимо ручного экспорта, бот сам присылает владельцу свежий архив (с тегом `#backup`) в четырёх случаях: при каждой подписке/отписке (заодно — уведомление, кто именно, со ссылкой на профиль), при смене квартала, раз в неделю и перед плановой остановкой (`SIGTERM` при редеплое — финальный снапшот, заодно сигнал «бот гасится»). На эфемерных хостах без тома на `/data` это единственный способ не потерять список подписчиков между редеплоями.

Shutdown-бэкап — подстраховка в дополнение к событийным, а не замена: он ловит плановую остановку, но не аварийную (`SIGKILL`/нехватка памяти/слишком короткий grace-период). Поэтому событийные бэкапы (по подписке/ротации) остаются основным механизмом, а shutdown добирает «последнюю милю».

> 💡 Доставка авто-бэкапов идёт владельцу напрямую по `OWNER_ID`, поэтому владелец должен хотя бы раз нажать `/start` — иначе Telegram не даст боту написать первым (та же причина, по которой важно быть подписанным; см. раздел запуска).

## Рассылка сообщений подписчикам

1. Напиши боту `/broadcast`.
2. Пришли сообщение для рассылки — поддерживаются текст, фото, видео, GIF, стикер, документ, голосовое.
3. Бот покажет превью и предложит две кнопки: **📢 Отправить** или **❌ Отмена**.
4. Если передумал до отправки превью — напиши `/cancel`.

## Стек

- [aiogram 3.x](https://docs.aiogram.dev/) — Telegram Bot API
- [aiohttp](https://docs.aiohttp.org/) — HTTP-клиент для Shikimori API и сервер healthcheck
- [Shikimori API](https://shikimori.io/api/doc) — история, списки и избранное (REST) + метаданные тайтлов (GraphQL)
- [Docker](https://www.docker.com/) — контейнеризация для деплоя

## Разработка и тестирование

Установка зависимостей разработки:

```bash
pip install -r requirements.txt -r requirements-dev.txt
```

Запуск тестов:

```bash
pytest tests/                        # все тесты
pytest tests/test_notify.py -v       # конкретный файл, подробный вывод
pytest tests/ -v --cov               # с измерением покрытия
```

**Важно:** тесты требуют переменные окружения `BOT_TOKEN` и `OWNER_ID` — они автоматически выставляются в `tests/conftest.py`, отдельно настраивать ничего не нужно.
