Telegram-бот, который следит за аниме и мангой любого пользователя на [Shikimori](https://shikimori.io) и отправляет весёлые уведомления подписчикам.
Достаточно указать ник отслеживаемого пользователя и своё отображаемое имя в настройках — бот готов к работе.

## Что умеет

- 📡 Проверяет историю и избранное пользователя каждые **15 минут**
- 🎬 Реагирует на события: добавил в список, начал смотреть/читать, пересматривает/перечитывает, бросил, завершил, поставил оценку, изменил оценку
- ⭐ Отслеживает новые добавления в избранное: аниме, мангу, персонажей и людей индустрии
- ✅ Поддерживает **аниме** (TV, фильм, OVA, ONA) и **мангу** (кроме ваншотов и додзинси)
- 📊 Собирает **квартальную статистику** просмотренного и прочитанного: жанры, темы, студии, издатели, источники адаптации, годы выпуска, оценки и сравнение с рейтингом Shikimori
- 🗓️ В начале нового квартала автоматически присылает владельцу **развёрнутый отчёт** с достижениями и сравнением с прошлым кварталом
- 👥 Система подписки — каждый подписывается сам командой `/start`
- 📋 Команда `/status` показывает что сейчас смотрит и читает пользователь
- 📈 Команда `/stats` — статистика за текущий квартал, `/stats all` — за всё время
- 📢 Команда `/broadcast` позволяет владельцу написать подписчикам напрямую
- 💾 Команды `/export` и `/import` для переноса списка подписчиков между хостингами
- 🩺 HTTP-эндпоинт `/health` для healthcheck — совместимость с хостингами, требующими открытый порт, и watchdog-перезапуск при зависании
- ⚠️ При ошибках фоновой проверки пишет в лог и уведомляет владельца в Telegram с антиспамом

## Команды

| Команда | Описание |
|---|---|
| `/start` | Подписаться на уведомления |
| `/stop` | Отписаться |
| `/status` | Что сейчас смотрит и читает отслеживаемый пользователь |
| `/stats` | Статистика за текущий квартал |
| `/stats all` | Агрегированная статистика за всё время |
| `/broadcast` | Написать сообщение подписчикам *(только для владельца)* |
| `/cancel` | Отменить текущую операцию *(только для владельца)* |
| `/subs` | Список подписчиков *(только для владельца)* |
| `/export` | Скачать `subscribers.json` *(только для владельца)* |
| `/import` | Загрузить `subscribers.json` из файла *(только для владельца)* |

## Статистика и отчёты

Бот ведёт статистику активности отслеживаемого пользователя и строит по ней отчёты.

**Источники данных.** Пользовательские данные (что просмотрено/прочитано, оценки, прогресс) берутся из публичного экспорта списков Shikimori (`/{user}/list_export/animes.json` и `mangas.json`) — без авторизации. Метаданные тайтлов (жанры, темы, студии, издатели, длительность, источник адаптации, год) подтягиваются через GraphQL API батчами, только для новых тайтлов.

**Что считается.** Завершённое и брошенное, средняя оценка и её распределение, топ по оценкам, распределение по жанрам/темам/демографии, студии (для аниме) и издатели (для манги), источники адаптации, возрастные рейтинги, хронология по годам выпуска, примерное время просмотра (для аниме) и количество глав (для манги), а также сравнение личных оценок со средним рейтингом Shikimori.

**Команда `/stats`** показывает статистику за текущий квартал, разбитую на сообщения по темам (аниме / манга). `/stats all` — агрегированную статистику за всё время. Обе команды доступны всем подписчикам и читают данные из локальных файлов, не нагружая API.

**Квартальный отчёт** формируется автоматически при смене квартала (1 января, апреля, июля, октября) и отправляется владельцу. Он включает полную статистику за квартал, сравнение с предыдущим кварталом и блок «достижений». Если бот был выключен в момент смены квартала — отчёт сформируется при следующем запуске (с защитой от повторной отправки).

При первом запуске в середине квартала статистика за текущий квартал будет неполной — об этом честно сообщается в заголовке (`статистика с DD.MM.YYYY`). Полная статистика за всё время (`/stats all`) при этом строится сразу и корректно, так как берётся из экспорта списков, а не из событий.

## Установка

### 1. Клонировать репозиторий

```bash
git clone https://github.com/WorgaNomoR/ShikiUpdatesBot.git
cd ShikiUpdatesBot
```

### 2. Установить зависимости

```bash
pip install -r requirements.txt
```

### 3. Создать бота

Напиши [@BotFather](https://t.me/BotFather) → `/newbot` → получи токен.

### 4. Узнать свой Telegram ID

Напиши [@userinfobot](https://t.me/userinfobot) — он пришлёт твой числовой ID.

### 5. Задать переменные окружения

```bash
export BOT_TOKEN="токен_от_BotFather"
export OWNER_ID="твой_telegram_id"
```

Или создай файл `.env` (см. `.env.example`) и загрузи его через `python-dotenv`.

### 6. Запустить

```bash
python main.py
```

## Запуск через Docker

Альтернативный способ запуска — через Docker-контейнер. Это упрощает развёртывание и гарантирует одинаковую среду выполнения на любом сервере.

### 1. Установить Docker

Убедись, что у тебя установлен [Docker](https://docs.docker.com/get-docker/) и [Docker Compose](https://docs.docker.com/compose/install/).

### 2. Создать файл `.env`

Создай файл `.env` в корне проекта со следующим содержимым:

```
BOT_TOKEN=токен_от_BotFather
OWNER_ID=твой_telegram_id
```

### 3. Запустить через docker-compose

```bash
# Собрать и запустить контейнер
docker-compose up -d --build

# Просмотреть логи
docker-compose logs -f

# Остановить контейнер
docker-compose down
```

### 4. Обновление

При изменениях в коде:

```bash
# Пересобрать контейнер
docker-compose build

# Перезапустить
docker-compose up -d
```

### Примечания

- По умолчанию данные бота хранятся в контейнере в `/data`.
- В `docker-compose.yml` папка `./data` на хосте монтируется в `/data` внутри контейнера.
- Для работы в продакшене достаточно только `requirements.txt` — тестовые зависимости (`requirements-dev.txt`) не включаются в рабочий образ.

## Healthcheck

Бот поднимает лёгкий HTTP-сервер параллельно с основным циклом (модуль `healthcheck.py`). Он нужен для двух целей:

- **Совместимость с хостингами**, которые требуют открытый порт или healthcheck-эндпоинт (Render, Fly.io, Cloud Run и подобные). Без него такие платформы могут считать процесс «не стартовавшим».
- **Watchdog**: эндпоинт `/health` отдаёт `200` только если бот «пульсирует» — успешно завершил итерацию фонового цикла недавно. Если пульс протух (цикл завис или умер) — отдаётся `503`, и хостинг с настроенным healthcheck перезапустит контейнер.

| Эндпоинт | Ответ |
|---|---|
| `GET /health` | `200` если бот живёт, `503` если последний успешный цикл был давно |
| `GET /` | `200` — простой признак, что порт открыт |

**Порог живости** адаптивный — три пропущенных цикла (`CHECK_INTERVAL × 3`, по умолчанию 45 минут). **Порт** берётся из переменной окружения `PORT` (хостинги обычно прокидывают её сами), по умолчанию `8080`. В настройках хостинга в качестве Healthcheck Path укажи `/health`.

Дополнительных зависимостей не требуется — используется серверная часть уже установленного `aiohttp`.

## Конфигурация

Основные параметры в начале `main.py`:

| Константа | По умолчанию | Описание |
|---|---|---|
| `BOT_TOKEN` | из переменной окружения | Токен Telegram-бота от BotFather |
| `OWNER_ID` | из переменной окружения | Telegram ID владельца для админ-команд и уведомлений об ошибках |
| `SHIKI_USER` | `WNR` | Ник на Shikimori |
| `DISPLAY_NAME` | `Ворга` | Отображаемое имя в сообщениях |
| `SHIKI_BASE_URL` | `https://shikimori.io` | Домен — менять при смене зеркала |
| `CHECK_INTERVAL` | `900` (15 мин) | Интервал проверки в секундах |
| `ERROR_NOTIFY_INTERVAL` | `1800` (30 мин) | Минимальная пауза между уведомлениями владельцу об ошибках |
| `DATA_DIR` | `/data` | Папка для файлов состояния; задаётся через переменную окружения |
| `PORT` | `8080` | Порт healthcheck-сервера; задаётся через переменную окружения |

## Поведение при ошибках

Фоновый цикл проверки истории и избранного не останавливается из-за единичной ошибки Shikimori API, сети, парсинга или отправки сообщения. Бот:

- пишет полный traceback в лог;
- пробует отправить владельцу короткое сообщение об ошибке в Telegram;
- не отправляет такие уведомления чаще одного раза в `ERROR_NOTIFY_INTERVAL`;
- продолжает работу и повторяет проверку после `CHECK_INTERVAL`.

Сбор статистики устроен так же отказоустойчиво: недоступность экспорта или GraphQL не роняет бота — отчёт просто выйдет без обогащённых метаданных, а битый файл состояния заменяется пустой структурой. Команды `/stats` не делают сетевых запросов и не могут упасть из-за недоступности Shikimori.

Если процесс Python полностью завершился, сервер выключился или Telegram недоступен, уведомление отправить не получится — в таких случаях нужны логи хостинга или внешний мониторинг (для последнего и пригодится healthcheck-эндпоинт).

## Файлы данных

Бот хранит файлы состояния — **не коммитить** (они уже в `.gitignore`):

- `seen_ids.json` — ID уже обработанных событий истории
- `subscribers.json` — список подписчиков
- `seen_favourites.json` — ID уже обработанных записей избранного
- `stats_all.json` — статистика за всё время: архив тайтлов и агрегаты
- `stats_current.json` — события текущего квартала
- `quarters/` — замороженные снапшоты завершённых кварталов

По умолчанию файлы создаются в каталоге `/data`. Если нужно хранить их в другом месте, укажи путь через переменную окружения:

```
DATA_DIR=/путь/к/папке
```

При первом запуске бот молча запоминает текущую историю и избранное, не спамит прошлыми событиями — отслеживает только новые.

## Перенос подписчиков при смене хостинга

1. Напиши боту `/export` — он пришлёт `subscribers.json` в чат
2. На новом хостинге запусти бота
3. Отправь боту файл с подписью `/import` в поле caption

## Рассылка сообщений подписчикам

1. Напиши боту `/broadcast`
2. Пришли сообщение для рассылки — поддерживаются текст, фото, видео, GIF, стикер, документ, голосовое
3. Бот покажет превью и предложит две кнопки: **📢 Отправить** или **❌ Отмена**
4. Если передумал до отправки превью — напиши `/cancel`
